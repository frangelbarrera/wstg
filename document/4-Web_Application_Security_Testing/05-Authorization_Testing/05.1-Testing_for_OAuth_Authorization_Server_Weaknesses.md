# Pruebas de Debilidades del Servidor de Autorización de OAuth

## Resumen

OAuth almacena las identidades de los usuarios y sus correspondientes derechos de acceso con el Servidor de Autorización (AS, por sus siglas en inglés). El AS juega un rol crucial durante el flujo de OAuth ya que otorga a los clientes acceso a recursos. Para poder hacer eso de manera segura, debe validar apropiadamente parámetros que son parte del flujo de OAuth.

Fallar en validar los parámetros podría llevar a takeover de cuenta, acceso no autorizado a recursos y elevación de privilegios.

## Objetivos de Prueba

- Identificar debilidades en el Servidor de Autorización.

## Cómo Probar

Para probar las debilidades del AS, apuntarás a:

1. Recuperar credenciales usadas para autorización.
2. Otorgarte acceso a recursos arbitrarios a través de navegación forzada.
3. Evadir la autorización.

### Probar Validación Insuficiente de Redirect URI

Si el `redirect_uri` no se valida apropiadamente, se puede diseñar un enlace que contiene una URL apuntando a un servidor controlado por un atacante. Esto puede usarse para engañar al AS para que envíe un código de autorización al atacante. En el siguiente ejemplo, `client.evil.com` se usa como el `redirect_uri` forjado.

```text
https://as.example.com/authorize?client_id=example-client&redirect_uri=http%3A%2F%client.evil.com%2F&state=example&response_mode=fragment&response_type=code&scope=openid&nonce=example
```

Si un usuario abre este enlace en el user agent, el AS redirigirá el user agent a la URL maliciosa.

Un atacante puede capturar el valor `code` pasado en la URL suplantada y luego enviarlo al endpoint de token del AS.

La siguiente solicitud ilustra una solicitud de autorización que envía el `redirect_uri` al servidor de autorización. El cliente `client.example.com` envía una solicitud de autorización al AS `as.example.com` con el redirect URI codificado en URL `http%3A%2F%2Fclient.example.com%2F`.
```http
GET /authorize
    ?redirect_uri=http%3A%2F%2Fclient.example.com%2F
    &client_id=example-client
    &errorPath=%2Ferror
    &scope=openid%20profile%20email
    &response_type=code
    &response_mode=query
    &state=example
    &nonce=example
    &code_challenge=example
    &code_challenge_method=S256 HTTP/1.1
Host: as.example.com
```

El AS responde con una redirección que contiene el código de autorización. Esto puede intercambiarse con un token de acceso en la solicitud de token. Como se muestra abajo, la URL en el encabezado `Location` es la URI dada en el parámetro `redirect_uri` previo.

```http
HTTP/1.1 302 Found
Date: Mon, 18 Oct 2021 20:46:44 GMT
Content-Type: text/html; charset=utf-8
Content-Length: 340
Location: https://client.example.com/?code=example&state=example
```

Para probar si el AS es vulnerable a validación insuficiente de redirect URI, capturar el tráfico con un proxy de intercepción HTTP tal como ZAP.

1. Iniciar el flujo de OAuth y pausarlo en la solicitud de autorización.
2. Cambiar el valor del `redirect_uri` y observar la respuesta.
3. Investigar la respuesta e identificar si el parámetro `redirect_uri` arbitrario fue aceptado por el AS.

Si el AS redirige el user agent al `redirect_uri` que especificaste, el AS no valida apropiadamente el `redirect_uri`.

Adicionalmente, ver la sección `Common Filter Bypass` en [Pruebas de Server-Side Request Forgery](../07-Input_Validation_Testing/19-Testing_for_Server-Side_Request_Forgery.md) para identificar evasiones comunes para validación de redirect URI.

### Probar Inyección de Código de Autorización

Durante el intercambio de código del flujo de Authorization Code, un código es emitido por el AS al cliente y luego intercambiado contra el endpoint de token para recuperar un token de autorización y un token de refresco.

Conducir las siguientes pruebas contra el AS:

1. Enviar un código válido para otro `client_id`.
2. Enviar un código válido para otro propietario de recurso.
3. Enviar un código válido para otro `redirect_uri`.
4. Reenviar el código más de una vez (replay de código).

#### Probar Clientes Públicos

La solicitud enviada al endpoint de token contiene el código de autorización. Se intercambia contra el token. Capturar esta solicitud con un proxy de intercepción HTTP como ZAP y reenviar la solicitud con valores modificados.

```http
POST /oauth/token HTTP/1.1
Host: as.example.com
[...]

{
    "errorPath":"/error",
    "client_id":"example-client",
    "code":"INJECT_CODE_HERE",
    "grant_type":"authorization_code",
    "redirect_uri":"https://client.example.com"
}
```

Si el AS responde con un `access_token`, el código se inyectó exitosamente.

#### Probar Clientes Confidenciales

Como el flujo de OAuth para clientes confidenciales está adicionalmente protegido por un client secret, no es posible enviar directamente un código de autorización al endpoint de token. En su lugar, inyectar el código de autorización en el cliente. Este código inyectado entonces se enviará en la solicitud de token, emitida por el cliente confidencial junto con el client secret.

Primero, capturar un código de autorización del AS:

1. Iniciar el flujo de authorization code con la usuaria Alice. Pausar cuando recibas un código del AS.
2. No enviar el código al cliente y tomar nota del código y estado correspondiente.

Luego, inyectar el código:

 1. Iniciar el flujo de authorization code con la usuaria Mallory e inyectar el código previamente recolectado y valores de estado para la usuaria Alice en el proceso.
 2. Cuando el ataque sea exitoso, el cliente debería ahora estar en posesión de un `authorization_token` que otorga acceso a recursos propiedad de la usuaria Alice.

```http
GET /callback?code=INJECT_CODE_HERE&state=example HTTP/1.1
Host: client.example.com
[...]

```

### Probar Ataque de Downgrade de PKCE

Bajo ciertas circunstancias la extensión PKCE puede removerse del flujo de authorization code. Esto tiene el potencial de dejar a los clientes públicos vulnerables a ataques mitigados por la extensión PKCE.

Esto puede ocurrir cuando:

- El AS no soporta PKCE.
- El AS no valida propiamente PKCE.

Ambos pueden probarse con un proxy de intercepción HTTP como ZAP. Conducir las siguientes pruebas:

1. Enviar la solicitud de autorización sin el parámetro `code_challenge=sha256(xyz)` y `code_challenge_method`.
2. Enviar la solicitud de autorización con un valor vacío para el parámetro `code_challenge=sha256(xyz)`.
3. Enviar la solicitud de autorización con un valor forjado para el parámetro `code_challenge=sha256(xyz)`

El ejemplo a continuación resalta los valores a modificar:

```http
GET /authorize
    ?redirect_uri=http%3A%2F%client.example.com
    &client_id=example-client
    &errorPath=%2Ferror
    &scope=openid%20profile%20email
    &response_type=code
    &response_mode=web_message
    &state=example-state
    &nonce=example-nonce
    &code_challenge=MODIFY_OR_OMIT_THIS
    &code_challenge_method=MODIFY_OR_OMIT_THIS
    &prompt=none HTTP/1.1
Host: as.example.com
[...]

```

El AS debería verificar el valor `code_verifier` en el intercambio de token. Para probar:

1. Enviar la solicitud de token sin el `code_verifier`.
2. Enviar la solicitud de token con un `code_verifier` vacío.
3. Enviar la solicitud de token con un `code_verifier` válido para un código de autorización diferente.

```http
POST /oauth/token HTTP/1.1
Host: as.example.com
[...]

{
"client_id":"example-client",
"code_verifier":"MODIFY_OR_OMIT_THIS",
"code":"example",
"grant_type":"authorization_code",
"redirect_uri":"https://client.example.com"
}
```

### Probar Cross-Site Request Forgery de Página de Consentimiento

Los ataques CSRF se describen en [CSRF](../06-Session_Management_Testing/05-Testing_for_Cross_Site_Request_Forgery.md). OAuth puede ser atacado con CSRF.

Para prevenir ataques CSRF OAuth, aprovecha el parámetro `state` como un token anti-CSRF.

Otras medidas pueden prevenir ataques CSRF también. El flujo PKCE mitiga CSRF. Un valor `nonce` puede actuar como un token anti-CSRF también.

Probar cada solicitud que contenga uno de los parámetros anti-CSRF usados por OAuth de acuerdo con las pruebas descritas en los casos de prueba de [CSRF](../06-Session_Management_Testing/05-Testing_for_Cross_Site_Request_Forgery.md).

La página de consentimiento se muestra a un usuario para verificar que este usuario consiente en que el cliente acceda al recurso en nombre del usuario. Atacar la página de consentimiento con CSRF podría otorgar a un cliente arbitrario acceso a un recurso en nombre del usuario. Los pasos de este flujo son:

1. El Cliente genera un parámetro state y lo envía con la solicitud de consentimiento.
2. El User Agent muestra la página de consentimiento.
3. El Propietario del Recurso otorga acceso al Cliente.
4. El consentimiento se envía al AS junto con los scopes reconocidos.

Usar un proxy de intercepción HTTP como ZAP para probar si el parámetro state se valida apropiadamente.

```http
POST /u/consent?state=Tampered_State HTTP/1.1
Host: as.example.com
[...]

state=MODIFY_OR_OMIT_THIS
&audience=https%3A%2F%2Fas.example.com%2Fuserinfo
&scope%5B%5D=profile
&scope%5B%5D=email
&action=accept
```

### Probar Clickjacking

El clickjacking se describe en [Pruebas de Clickjacking](../11-Client-side_Testing/09-Testing_for_Clickjacking.md). Cuando la página de consentimiento es propensa a clickjacking y el atacante está en posesión del `client_id` (para clientes públicos, o el client secret para clientes confidenciales), el atacante puede forjar el consentimiento del usuario y ganar acceso al recurso solicitado a través de un cliente rogue.

#### Cómo Probar

Para que este ataque sea exitoso, el atacante necesita cargar la página de autorización en un iframe.

La siguiente página HTML puede usarse para cargar la página de autorización en un iframe:

```html
<html>
    <head>
        <title>Clickjack test page</title>
    </head>
    <body>
        <iframe src="https://as.example.com/auth/realms/example/login-actions/required-action?execution=OAUTH_GRANT&client_id=example-client" width="500" height="500"></iframe>
    </body>
</html>
```

Si se carga exitosamente, el sitio es vulnerable a clickjacking.

Ver [Pruebas de Clickjacking](../11-Client-side_Testing/09-Testing_for_Clickjacking.md) para una descripción detallada de cómo tal ataque puede conducirse.

### Probar Tiempo de Vida del Token

OAuth tiene dos tipos de tokens: el token de acceso y el token de refresco. Un token de acceso debería estar limitado en la duración de su validez. Eso significa que es de corta duración: una buena duración depende de la aplicación y podría ser de 5 a 15 minutos.

El token de refresco debería ser válido por una duración más larga. Debería ser un token de un solo uso que se reemplaza cada vez que se ha usado.

#### Probar Validación de Tiempo de Vida del Token de Acceso

Cuando se usa un JSON Web Token (JWT) como token de acceso, es posible recuperar la validez del token de acceso del JWT decodificado. Esto se describe en [Pruebas de JSON Web Tokens](../06-Session_Management_Testing/10-Testing_JSON_Web_Tokens.md). Es posible que el AS no valide apropiadamente el tiempo de vida del JWT.

Para probar el tiempo de vida del token de acceso, usar un proxy de intercepción HTTP tal como ZAP. Intercepta una solicitud a un endpoint que contiene un token de acceso. Poner esta solicitud en el repeater y dejar pasar el tiempo objetivo. La validez de un token de acceso debería estar entre 5 y 15 minutos, dependiendo de la sensibilidad de los recursos.

Tales solicitudes podrían verse como el siguiente ejemplo. El token también podría transportarse de otras maneras, por ejemplo, en una cookie.

```http
GET /userinfo HTTP/1.1
Host: as.example.com
[...]
Authorization: Bearer eyJhbGciOiJkaXIiL[...]

```

Probar la validación de tiempo de vida enviando la solicitud después de que hayan pasado varias longitudes de tiempo, por ejemplo, después de 5 minutos, 10 minutos, y 30 minutos.

Este proceso puede optimizarse automatizando los pasos y el logging de la respuesta del servidor. Cuando se recibe una respuesta de estado HTTP 403 (en lugar de estado HTTP 200), esto puede indicar que el token de acceso ya no es válido.

#### Probar Validación de Tiempo de Vida del Token de Refresco

Los tokens de refresco tienen un período de validez más largo que los tokens de acceso. Debido a su larga validez, deberían invalidarse después de que se usan en un intercambio contra un token de acceso.

Los tokens de refresco se emiten en la misma solicitud de token donde el token de acceso se entrega al cliente.

Usar un proxy de intercepción HTTP tal como ZAP. Configurar la prueba haciendo lo siguiente:

1. Recuperar un token de refresco válido.
2. Capturar la solicitud que se usa para intercambiar el token de refresco contra un nuevo token de acceso.
3. Enviar la solicitud capturada al repeater de solicitudes.

En el siguiente ejemplo, el token de refresco se envía como parte del cuerpo POST.

```http
POST /token HTTP/1.1
Host: as.example.com
Cookie: [...]
[...]

grant_type=refresh_token
&refresh_token=eyJhbGciOiJIUz[...]
&client_id=example-client

```

Conducir las siguientes pruebas:

1. Enviar el token de refresco y determinar si el AS entrega un token de acceso.
2. Repetir los pasos con el mismo token de refresco para evaluar cuán a menudo se acepta un solo token de refresco.

Cuando se usa un JWT como token de refresco, es posible recuperar la validez del token de refresco del JWT decodificado. Esto se describe en [Pruebas de JSON Web Tokens](../06-Session_Management_Testing/10-Testing_JSON_Web_Tokens.md). El token de refresco podría ser válido por un período de tiempo más largo, pero debería tener una fecha de expiración.

Seguridad adicional puede ganarse con un mecanismo de detección de robo. Si un token de refresco se usa en un intercambio de token más allá de su validez (o tiempo de vida), el AS invalida todos los tokens de refresco. Para probar este mecanismo:

1. Enviar el token de refresco y determinar si el AS entrega un token de acceso.
2. Repetir los pasos con el mismo token de refresco hasta que se invalide.
3. Usar el token de refresco de la última respuesta de token

Si todos los tokens de refresco que se emitieron al cliente para este propietario de recurso se invalidan, el AS tiene detección de robo de token.

## Remediación

La mayoría de los ataques contra OAuth AS pueden mitigarse validando la existencia y contenido de parámetros durante el código de autorización e intercambio de token.

Restringir el lapso de tiempo y uso permitido para credenciales tales como el código de autorización y token de refresco. Esto puede mitigar algunos tipos de ataques y también limita el uso de tales credenciales para un atacante, si se ganan.

La configuración apropiada de mitigación de seguridad como CORS, tokens anti-CSRF, y encabezados anti-clickjacking puede mitigar o limitar el impacto de ataques.

- Siempre validar si todos los parámetros están presentes, y validar sus valores.
- Usar la extensión PKCE para asegurar apropiadamente el código de autorización e intercambio de token.
- No permitir fallback para características de seguridad como la extensión PKCE.
- Restringir el tiempo de vida de credenciales.
- Usar credenciales solo una vez donde sea posible, por ejemplo, el código de autorización.
- Configurar mitigación de seguridad disponible como CORS, tokens anti-CSRF, y encabezados anti-clickjacking.
