# Pruebas de Self DOM Based Cross-Site Scripting

## Resumen

Self DOM-Based Cross-Site Scripting es un ataque específico y necesita conocimiento previo de cross site scripting basado en DOM e ingeniería social exitosa. El término 'self' (propio) es una referencia aquí al hecho de que el usuario necesita inyectar el payload en el campo de entrada, y así ejecutar la vulnerabilidad por sí mismo. La vulnerabilidad es además específica, ya que la Política de Seguridad de Contenido (CSP, por sus siglas en inglés) del sitio web puede bloquear la ejecución de scripts.

Este escenario usará el término "sink" de la siguiente manera: En computación, un sink, event sink o data sink es una clase o función diseñada para recibir entrada o eventos de otro objeto o función. Así, para encontrar posibles vulnerabilidades primero necesitamos identificar los sinks de la aplicación que queremos probar.

## Cómo Probar

El proceso de prueba para cross site scripting basado en Self DOM sigue:

1. Buscar sinks vulnerables, que permitan entrada del usuario.
2. Una vez que se identifica un posible sink, se puede insertar un payload.
3. Verificar el log de errores en las herramientas de desarrollador del navegador para ver el resultado, y sacar conclusiones.
4. Verificar si un atacante podría convencer a un usuario de insertar el payload sin requerir conocimiento técnico extenso.

### Ejemplo

Este ejemplo específico es de este [ticket de HackerOne](https://hackerone.com/reports/406587).

En el ejemplo, la siguiente función JavaScript se ejecuta en el sitio web `https://example.com`.

```js
//Marketo Form Code
function strip(html) {
    var tmp = document.createElement("DIV");
    tmp.innerHTML = html;
    return tmp.textContent || tmp.innerText || "";
}

$('form').submit(function() {
    $('textarea').val(function() {
        return strip($(this).val());
    });
});
```

El abuso de esta funcionalidad puede describirse de la siguiente manera:

1. El manejador del evento `submit` pasa el valor actual de cualquier elemento `textarea` a la función `strip`.
2. Esta función crea un nuevo elemento `div` y establece la propiedad `innerHTML` al valor proporcionado.
3. En el último paso luego devuelve la propiedad `textContent` del `div` resultante.

Este tipo de código se usa típicamente para eliminar etiquetas HTML de una cadena, ya que la propiedad `textContent` contiene la cadena que fue renderizada por el navegador cuando se analizó el HTML. Este método particular es inherentemente inseguro porque usa `innerHTML`. Cuando se proporciona entrada del usuario a la propiedad `innerHTML`, es analizada por el navegador web y por lo tanto puede llevar a la ejecución de JavaScript malicioso.

El siguiente payload puede ser usado para probar la vulnerabilidad: `<img src=x onerror=alert(1) />`

La consola del desarrollador mostraría dos errores: Uno que indica que `https://www.example.com/x` fue solicitado y devolvió un 404 (debido al atributo src de la etiqueta img). Otro que reporta una violación del CSP del sitio web.

Este segundo error ocurrió porque el navegador intenta ejecutar el código JavaScript en el atributo `onerror`, pero el CSP del sitio web previno la ejecución. Realizar las mismas acciones en un navegador con CSP deshabilitado permitió que el JavaScript en el atributo `onerror` se ejecutara.

Un atacante podría explotar esta vulnerabilidad convenciendo a un usuario de pegar un payload malicioso en el campo 'message' del formulario de contacto y luego hacer clic en el botón "send message". Este ataque podría ser mejorado convenciendo al usuario de usar una versión del navegador que no soporte CSP.
